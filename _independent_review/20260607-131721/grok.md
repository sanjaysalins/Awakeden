# Independent review — grok (OK, 113s)

**Critique of PIPELINE_HARDENING_PLAN.md (v4 — REUSE-FIRST)**

The plan correctly names the two root causes: (1) no forced set-level or catalogue check before lock, and (2) Scripture trusted from a corrupt cache (data/kjv_cache.json missing the Ps 22:7 comma) with no independent witness. The intent to make CTA-clause and hook-opener repetition blocking, to use punctuation-strict comparison, and to rebuild the registry from disk rather than append-only are direct responses to the documented failures of short_gate.gate_g7_freshness (non-blocking landing fingerprint on thread-slug only, punctuation-blind norm()).

**Core contradiction on reuse vs. duplication.**  
The title and repeated language claim “REUSE-FIRST” and “Do NOT edit the sibling-repo `short_gate` or `per_turn_synth` (reuse-don’t-duplicate).” The actual mechanism is “Port the proven logic into this repo’s `pipeline/` and wrap it” plus “build the gate in THIS repo by PORTING short_gate’s proven pieces (corpus, verbatim checker, freshness registry, sha-stamp)”. This is a fork, not reuse. The ported check_quote / verses_for / quotes_from_narration / landing_fingerprint / register_episode logic now lives in two places. No sync process, no import of the original as a library, no vendoring manifest with hash, and no test that asserts the punctuation-strict mode produces equivalent results to the source on the source’s own fixtures. Future fixes or alias improvements in `…\PythonProject1\jesus\narration\short_gate\kjv.py` will not flow here. This is the exact duplication concern the brief requires examination of.

**Chokepoint is narrower than the actual execution surface.**  
The plan designates `handoff.run_audio_pipeline` (called by runner.py, _finalize.py, _fix26_audio.py) as the lock enforcement point and states that direct invocation of the foreign per_turn_synth is “a documented, discouraged bypass (named, not silently possible).” This does not close root cause #1. The synthesis code that actually emits audio lives in the sibling and already exposes --no-gate. cli_pipeline.py, ad-hoc scripts, manual python invocations, and any caption/timing path that does not transit the exact handoff function remain outside the guard. Declaring one function “the sole sanctioned audio entry” and writing a .locked file with a spoken-text hash does not prevent someone from operating on a folder that never had a .locked or whose .locked is stale. The enforcement inventory is incomplete on its face.

**Cross-repo data management introduces new single points of fragility that are under-specified.**  
- Copying `C:\Users\sanjay\PycharmProjects\HF-POC\series\03_furgiven\tools\data\kjv.json` → `data/kjv_corpus.json` with “pin source path + provenance” creates a second source of truth. No process is described for detecting when the HF-POC source changes, which already-witnessed verses now differ, or how to force re-witnessing. An unwitnessed verse FAILs the lock (correct for integrity), but the operational drift risk is new and unmechanized.
- Registry rebuild “from on-disk .locked folders” is the right anti-phantom design. The glob description (“globbing both `longform/**/v1/**/narration.md` (this repo) and the engine `NARRATION_TREE_DIR`”) is imprecise — it should be keyed on .locked presence, not narration.md. Legacy folders with spaces (explicitly retained per the project’s loaded Claude.md) are at risk of being invisible to the glob. Engine-produced shorts in NARRATION_TREE_DIR were not produced under cli_lock.py; without an explicit backfill pass that walks the foreign tree and either creates .locked files or seeds the registry with provenance, the “across-catalogue” check will be incomplete for the first many runs. “Artifact slug” as the registry key is never defined (folder name? internal series+episode? normalized title?). Collisions or key instability on renames/legacy paths are unaddressed.

**Parser and veed_io decoupling assumptions are under-validated.**  
A1 claims `pipeline/narration_parse.py` (spoken_blocks, quoted_spans_with_refs, sentences) replaces the “broken veed_io/_extract_spoken.py (MOVEMENT-only → empty on shorts)”. The provided project file map does not surface a veed_io/ directory in this repo. If the extractor lives in the sibling, then “pipeline parses and passes plain spoken strings down to veed_io” and “no upward import” requires the actual call site that feeds the captioner to be changed in the right repo. The plan also lists the “veed_io caption/timing entry” as part of the enforcement inventory that “must check .locked”. If that entrypoint is not inside handoff or another file this repo owns, the lock cannot be enforced there. The claim that the 8 shorts + 2 longs “parse to non-empty spoken text” and that untagged quotes “resolve or are explicitly listed (not silently dropped)” is an acceptance criterion, not evidence that the extraction heuristics will be reliable without a manifest/scripture.json (which the hand-authored folders lack and which the original short_gate relied on).

**Acceptance criteria are narrow and risk both false negatives on new patterns and false positives on legitimate language.**  
A3 requires only that the ported+strengthened gate flag normalized “come to him” in 8/8 and the “thousand years” opener family on the known Psalm 22 cluster. This is necessary as a regression gate but insufficient as proof the mechanism is sound. “opener_family” (“thousand years” + similar stems) and the CTA normalized-substring logic are underspecified. CTA repetition is “never whitelisted”; only mid-body doctrinal phrases are. Core gospel phrasing that legitimately recurs across unrelated episodes (e.g., John 3:16 language, “believe in him”, “come to me”) will therefore trigger BLOCKING failures on future work, forcing either growing manual whitelists or overrides. No false-positive budget or example set is required before declaring the gate stable.

**New surface area and sequencing risks.**  
The plan adds cli_lock.py, verify_narration.py, cli_witness_verse.py, narration_parse.py, freshness.py, plus modifications to independent_review.py (new --type cluster + LENS_CLUSTER) and handoff.py, plus a new registry file and a B1 audit step. This is substantial new code for a problem whose primary real cost the plan itself correctly identifies as “rewriting 8 LOCKED shorts + re-panel”. The sequencing (A3 gates B/C) is sound, but the re-verification section does not address that material hook/CTA text changes in the 8 shorts will likely require re-running at least Phase A of visual (scene_plan) because the visual arc is deliberately mirrored to the narration thread spine. The cost section mentions audio and long-form caption/timing re-runs but is silent on visual re-work.

**Honest but incomplete admission of inert prior machinery.**  
C2 correctly flags that calibration.jsonl / panel_misses / append_record were never wired. Building another registry + chokepoint + verify CLI on top of the same culture without a committed writer path and owner for the new signals the cluster gate will produce risks creating the next piece of inert infrastructure.

**Minor but concrete gaps**
- “Sha-stamp lock … hashing the spoken text (not raw md)” is good, but the .locked file format, location relative to the folder, and what happens on a spoken-text edit after .locked exists are not specified.
- Rule-8 is declared “short-only”. The interaction between Rule-8, anchor-verse-quoted, and the cluster gate on long-form movements is not detailed.
- The plan requires the 8+2 to flow through cli_lock.py after the A2 rewrite “lets them pass cluster.” It does not require that the same artifacts also pass the full deterministic tier (B3 punctuation-strict KJV + Rule-8 + anchor) before the registry is backfilled; a post-rewrite KJV divergence could slip through if B is not forced before the final re-lock.

The plan has the right posture (fail-closed, proof on the actual miss first, separate manual witness tool, no touching foreign code, explicit honesty about the re-authoring cost). The execution design, however, converts a reuse opportunity into a fork, places the lock at a single owned function while the real synthesis surface is wider, and adds cross-boundary data (corpus copy + multi-repo glob) whose synchronization obligations are not mechanized. These are not cosmetic.

VERDICT: REVISE
TOP FIXES:
1. Stop calling a code copy “reuse.” Either expose the relevant short_gate pieces (kjv.py + gates.py core) as an importable package from the sibling (PYTHONPATH or proper packaging) so this repo can call the original check_quote / freshness logic and only add the punctuation-strict mode and CTA/opener fingerprints as parameters or extensions, or create a tiny shared verification package both repos depend on. If a true port is unavoidable, add an explicit vendoring step + a fixture test that runs the source short_gate functions and the ported versions on the same inputs and asserts behavioral equivalence (or documents the exact deltas introduced by strict mode).
2. Make the enforcement story match the actual surface instead of declaring one function canonical. Enumerate every entry point that can cause audio or caption output (handoff callers, cli_pipeline paths, any _fix*.py, direct synth invocations, veed_io feeders) and either put the .locked + spoken-hash check in each owned entry or require an explicit attestation token produced by cli_lock.py / verify_narration.py. For foreign scripts, document and enforce a thin pre-flight wrapper in this repo that refuses to launch them unless a current .locked matches the current spoken text. Do not rely on “documented, discouraged.”
3. Specify the registry and corpus mechanics with concrete globs, key derivation, legacy handling, and drift detection before any implementation. Define the exact slug key, glob for .locked (not narration.md), how spaced legacy folders and NARRATION_TREE_DIR episodes are discovered and attributed, and a required backfill pass that seeds real prior engine shorts so the cross-catalogue check is not empty on first use. Add a corpus provenance check (last source hash or mtime) and a diff command that lists witnessed verses now differing in the HF-POC source; treat divergence on a witnessed verse as a hard failure until re-witnessed via the separate cli_witness_verse.py.
