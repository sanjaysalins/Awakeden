# Independent review — gemini (OK, 44s)

This is an adversarial review of Phase C. The proposed lock chokepoint is fundamentally flawed, suffering from split-brain validation, blatant bypasses for the exact system that caused the original failure, and unimplemented checks. 

Here are the specific, critical failures:

**1. Split-Brain Validation (The Tagged File Bypass)**
There is a massive hash-binding gap in `pipeline/lock.py`. `run_lock()` hardcodes its validation to read the untagged file:
`md_path = folder / "narration.md"`
It runs KJV, Rule-8, and Cluster checks against this text. However, if it passes, it calls `write_lock(folder)` which calls `spoken_hash(folder)`. `spoken_hash()` delegates to `_spoken_source(folder)`, which explicitly prefers `narration-tagged.md`. 
**The Hole:** An invalid, misquoted, or templated `narration-tagged.md` will completely bypass validation if the base `narration.md` is clean. The script validates the clean file, but locks the hash of the invalid file. Since synth consumes the tagged file, the pipeline will happily render corrupted content with a valid `.locked` token.

**2. The Engine Exemption Defeats the Entire Purpose**
In `runner.py`: `code = handoff.run_audio_pipeline(folder, enforce_lock=False)`.
The stated context is that a standalone gate "reproduces the org failure that shipped the templated Psalm 22 cluster" and this lock is meant to stop it. That templated cluster was generated by the *orchestrator/engine*. By exempting the engine via `enforce_lock=False`, you have created a "fail-closed" gate that entirely ignores the automated pipeline. The system can still autonomously generate, render, and ship a bad cluster. The "pending narration-format reconciliation" excuse is a classic risk bypass; if the engine cannot pass the gate, it should not be allowed to render audio.

**3. The Catalogue Check is a Lie**
The docstring for `run_lock` claims it runs a "cross-artifact cluster check (Phase A) vs the folder's siblings + the catalogue." The code does not do this. It gathers `_sibling_folders(folder)` and calls:
`cluster_gate.cluster_check(arts, within_cluster=True)`
It checks siblings, but it **never** loads or checks `_REGISTRY` (`freshness_registry.json`). A newly generated short could perfectly duplicate a completed short from an entirely different batch, and this chokepoint will blindly pass it.

**4. Registry Race Conditions & Path Hardcoding**
In `pipeline/lock.py`:
* `register()` calls `rebuild_registry()` and writes directly to `_REGISTRY.write_text(...)`. There is zero file-locking. If a batch processor or user runs `cli_lock.py` concurrently on multiple folders, they will overwrite each other's registry states, leading to dropped tracking data.
* `rebuild_registry()` hardcodes its search to `_REPO.glob("longform/**/.locked")`. If content is generated or locked in `data/`, `_library/`, or a testing directory, it vanishes from the registry.
* Furthermore, `rebuild_registry()` hardcodes parsing `narration.md` to extract the `hook` and `cta`. If `narration-tagged.md` is the true spoken source being locked, the registry is caching stale metadata.

**5. False-Blocks on Missing Untagged Files**
If a folder legitimately only has `narration-tagged.md` (e.g. from an alternate workflow or cleanup), `_spoken_source` knows how to handle it, but `run_lock` fails immediately:
`if not md_path.is_file(): return {"ok": False, "folder": folder.name, "blocking": ["no narration.md"]}`.

VERDICT: FAIL
TOP FIXES:
1. Fix the split-brain in `run_lock`: it must evaluate `md_path = _spoken_source(folder)` and run all KJV/Rule-8/Cluster checks on the EXACT file that is being hashed and sent to synth.
2. Remove `enforce_lock=False` from `runner.py`; the engine must be forced to pass the validation gate before rendering audio, otherwise the original incident is still fully reproducible.
3. Implement the missing global registry check in `run_lock` (not just siblings) and add atomic file locks to `register()` to prevent concurrent wipeouts.
